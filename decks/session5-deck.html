Quick Recap — Session 7

Arrays vs. dictionaries

Session 7: Arrays are ordered lists — values accessed by numbered position. Use table.insert, table.remove, #table, and ipairs.

A dictionary table uses named keys instead of numbers — more like a data record than a list.

Array (Session 7)

-- Position-based
{ "Sword", "Shield", 45 }
-- item[1], item[2], item[3]

Dictionary (This Session)

-- Key-based
{ name="Sword", damage=45 }
-- item.name, item.damage

Building the Structure

Creating a dictionary table

Each entry is a key = value pair inside the curly braces.

local weapon = {
  name       = "Thornblade",
  damage     = 75,
  isMagic    = true,
  durability = 100,
  element    = "Nature"
}

This is a single variable — weapon — holding five named pieces of data. Think of it as a stat block for one item.

Style Tip

Aligning the = signs vertically makes the structure much easier to scan, especially with many keys.

Getting Values Out

Reading with dot notation

Access any key using a dot — the same syntax from Session 6. Those engine objects are dictionary tables under the hood.

local weapon = { name="Thornblade", damage=75, isMagic=true }

local wName   = weapon.name     -- "Thornblade"
local wDamage = weapon.damage   -- 75
local wMagic  = weapon.isMagic  -- true

Accessing a key that doesn't exist returns nil. So weapon.rarity would be nil above.

The Other Syntax

Bracket notation for keys

These two are equivalent for known key names:

local item = { name="Potion", value=30 }

local a = item.name
local b = item["name"]   -- identical

Bracket notation is essential when the key is stored in a variable:

local propName = "value"
local result   = item[propName]   -- same as item.value
When to use which

Dot notation when you know the key at write-time. Bracket notation when the key is in a variable or built dynamically.

Changing Values

Writing and adding properties

Assign to a key to update it, or create a new one — there's no separate "add" operation.

local enemy = { name="Sandcrawler", health=200, isAlive=true }

-- Update an existing property
enemy.health = enemy.health - 50   -- 150

-- Add a property that didn't exist before
enemy.isStunned   = true
enemy.stunDuration = 3

Clearing Data

Removing a property

Setting a key to nil removes it from the table.

local player = {
  name          = "Kaelthas",
  activeQuestId = 1042
}

-- Player abandoned the quest
player.activeQuestId = nil
-- key is gone; reading it returns nil
Common Pattern

Setting to nil is how you "turn off" or unset a property. Use this for clearing status effects, removing temporary states, and resetting optional data.

Looping Over All Keys

The pairs iterator

For dictionary tables, use pairs — it visits every key-value pair.

local stats = { strength=18, agility=12, intellect=9 }

for statName, statValue in pairs(stats) do
  local label = statName .. ": " .. statValue
end
Order Not Guaranteed

pairs does not visit keys in any predictable order. If order matters, use an array or maintain a separate ordered key list.

Going Deeper

Tables as property values

local npc = {
  name      = "Elder Morvain",
  isFriendly = true,
  rewards   = { "Ancient Scroll", "Gold Coin", "Blessing" },
  stats     = { level=40, wisdom=95 }
}

local firstReward  = npc.rewards[1]        -- "Ancient Scroll"
local npcLevel    = npc.stats.level         -- 40
local rewardCount = #npc.rewards            -- 3

Chain dot notation and brackets to drill as deep as needed.

Everything Together

Building and using an item record

function OnItemDrop(isBoss)
  local drop = {
    name       = "Embersteel Axe",
    damage     = 110,
    isMagic    = false,
    dropSource = nil
  }
  if isBoss then
    drop.damage     = drop.damage * 1.5
    drop.isMagic    = true
    drop.dropSource = "Boss"
  end
  return drop
end

Putting It All Together

Tables that manage themselves

So far we've stored data in tables and written separate functions to work with them. A cleaner pattern is to put the functions inside the table itself — so the data and the logic that manages it live in one place.

local RELICS_LABYRINTH = {
  relics = {},   -- the data lives here

  Register = function(self, unit) end,  -- add a relic
  Remove   = function(self, name) end,  -- remove a relic
  Edit     = function(self, name, key, val) end,  -- edit a property
  List     = function(self) end,         -- debug print all
}

Each function receives self as its first argument — the table passing itself in silently via the colon call syntax. That's how each function can reach self.relics to read and modify the shared data.

Helper Function: Add

Register — adding an entry

Each relic calls this from its own OnSpawn, passing itself in as unit. The function builds a record from the object's properties and appends it to the list.

Register = function(self, unit)
  local entry = {
    altarNum    = unit.altarNum,
    altarPoint  = unit.altarPoint,
    altarFacing = unit.altarFacing,
    relicName   = unit.relicName,
    relicStrID  = unit.relicStrID
  }
  table.insert(self.relics, entry)
end,   -- comma: more entries follow in the table

Called from each relic's OnSpawn:

function OnSpawn(self)
  RELICS_LABYRINTH:Register(self)
end
Remember

The global table must be initialized before any relic spawns. An empty RELICS_LABYRINTH = {} defined at load time is enough — the relics populate it themselves.

Helper Function: Remove

Remove — deleting an entry

To remove a specific entry, loop through the list to find it by a known property, then use table.remove with its index.

Remove = function(self, relicName)
  for i, relic in ipairs(self.relics) do
    if relic.relicName == relicName then
      table.remove(self.relics, i)
      return   -- stop once found and removed
    end
  end
end,

Notice we use i rather than _ here — we need the actual index to pass to table.remove. The return after removing stops the loop immediately so it doesn't keep searching.

Called like this:

RELICS_LABYRINTH:Remove("Thornblade")

Helper Function: Edit

Edit — changing a property

To update a specific property on a specific entry, find the entry by name, then assign the new value using bracket notation — since the key name is arriving as a variable.

Edit = function(self, relicName, propKey, newValue)
  for _, relic in ipairs(self.relics) do
    if relic.relicName == relicName then
      relic[propKey] = newValue   -- bracket notation: key is a variable
      return
    end
  end
end,

This is where bracket notation earns its place. You can't write relic.propKey here — that would look for a key literally named "propKey". The brackets tell Lua to use the value of the variable as the key name.

Called like this:

RELICS_LABYRINTH:Edit("Thornblade", "altarPoint", 5)
