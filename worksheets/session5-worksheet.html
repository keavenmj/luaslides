B

Trace the Function Call

Trace

Given this function, trace through the two calls and write what each returns.

function GetHealAmount(baseHeal, isBlessed)
  local amount = baseHeal
  if isBlessed then
    amount = amount * 2
  end
  return amount
end

local healA = GetHealAmount(40, false)
local healB = GetHealAmount(40, true)

Now trace this more complex call:

function RollEncounter(playerLevel, zoneDifficulty)
  local encounterLevel = zoneDifficulty + (playerLevel / 2)
  local isTough        = encounterLevel > 20
  return encounterLevel, isTough
end

local lvl, tough = RollEncounter(24, 10)

E

Write Your Own

Application

E1. Write a function called GetArmorClass that takes a baseArmor value and a boolean isShieldEquipped. If the shield is equipped, add 15 to the armor. Return the final armor class value.

E2. Write a function called CheckQuestCompletion that takes three parameters: killCount, killGoal, and hasDeliveredItem. The quest is complete when kills meet the goal AND the item has been delivered. Return a boolean indicating whether the quest is done, and a string message: either "Quest Complete!" or "Keep going."

✓ Answer Key — Session 5
A — Label the Parts
ScaleDamage = function name  |  baseDamage, scaleFactor = parameters  |  result = local variable  |  result (after return) = return value
Note that result appears in two roles here: it's a local variable and it becomes the return value. The function could also be written as return baseDamage * scaleFactor directly — same result, slightly more compact.
B1 — GetHealAmount Trace
healA = 40  |  healB = 80
First call: isBlessed is false, so the if block doesn't run. amount stays 40. Second call: isBlessed is true, so amount = 40 * 2 = 80. Same function, different outcome based on input.
B2 — RollEncounter Trace
lvl = 22  |  tough = true
encounterLevel = 10 + (24 / 2) = 10 + 12 = 22. Then isTough = 22 > 20 = true. Both values are returned and captured. The left variable lvl gets the first return value (22), and tough gets the second (true).
C1 — Returns 0
The function returns result, which is initialized to 0 and never updated. The calculation is done into damage (the parameter), but that's never stored back.
function ApplyResistance(damage, resistance)
  local result = damage * (1 - resistance)
  return result
end
C2 — Missing Return
The function never returns anything. name is assigned inside the function but the value is never sent back with return.
function GetBossName()
  local name = "Wraithguard"
  return name
end
D — Fill in Parameters
D1: params = playerLevel, requiredLevel  /  return = qualifies  |  D2: params = attackerName, victimName  /  return = msg
The parameters must match exactly what the function body uses — if the body references playerLevel and requiredLevel, those are the parameter names. The return value should be the variable that holds the computed result.
E1 — Sample Answer
function GetArmorClass(baseArmor, isShieldEquipped)
  local armor = baseArmor
  if isShieldEquipped then
    armor = armor + 15
  end
  return armor
end
Note: local armor = baseArmor creates a copy so we're not directly modifying the parameter. This is good practice — treat parameters as read-only inputs.
E2 — Sample Answer
function CheckQuestCompletion(killCount, killGoal, hasDeliveredItem)
  local isComplete = (killCount >= killGoal) and hasDeliveredItem
  local message
  if isComplete then
    message = "Quest Complete!"
  else
    message = "Keep going."
  end
  return isComplete, message
end
Both return values flow naturally from the logic. The caller can use both: local done, msg = CheckQuestCompletion(5, 5, true). If the caller only needs one value, they can capture just the first: local done = CheckQuestCompletion(5, 5, true).
