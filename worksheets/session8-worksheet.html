✓ Answer Key — Session 8
A — Property Reads
a = "Dragonscale Vest" | b = 170 | c = true | d = nil | e = nil
b: 85 * 2 = 170 — arithmetic works directly on a read property. d: setBonus exists as a key but its value is nil. e: rarity doesn't exist at all — also returns nil. Both behave identically when read.
B — Mutation Trace
name = nil | health = 180 | isAlive = true | isStunned = true | stunTimer = 3
health: 300 - 120 = 180.
isAlive: 180 > 0 is true — the comparison result is stored directly.
isStunned / stunTimer: These are new keys created by assignment — they didn't exist in the original table.
name: Set to nil, so the key is effectively removed.
C1 — Key Case Mismatch
Problem: the key is "Name" (capital N) but the read uses "name" (lowercase). Lua keys are case-sensitive.
Fixed read: local n = enemy.Name. Better practice: standardize all your keys to camelCase (name, health) and be consistent.
C2 — String in Numeric Loop
The "label" key holds a string. Adding a string to a number with + causes an error.
local total = 0
for _, v in pairs(stats) do
  if type(v) == "number" then
    total = total + v
  end
end
The built-in type() function returns the type of a value as a string. Using it as a guard lets the loop safely skip non-numeric values.
D — Nested Access
title = "The Sunken Keep" | objectives[2] = "Defeat the warden" | #objectives = 3 | reward.xp = 1200 | gold + xp = 1700
Chain notation drills into nested tables one level at a time. quest.reward.gold means: get the quest table, get its reward key (another table), then get that table's gold key.
E1 — Sample Answer
local spellData = {
  name      = "Frostbolt",
  manaCost  = 40,
  damage    = 60,
  slows     = true,
  cooldown  = 8
}

spellData.damage   = spellData.damage * 1.2
spellData.cooldown = nil
After the damage update: 60 * 1.2 = 72. Setting cooldown to nil removes the key entirely — reading spellData.cooldown afterwards returns nil.
E2 — Sample Answer
function SummarizeItem(item)
  local base = item.name .. " — " .. item.damage .. " dmg"
  if item.isMagic then
    return "[Magic] " .. base
  else
    return base
  end
end
Building base first and then conditionally prepending the tag keeps the two branches clean and avoids duplicating the damage string logic.
E3 — Sample Answer
function ApplyStatBuffs(character, buffs)
  for statName, buffValue in pairs(buffs) do
    if character[statName] ~= nil then
      character[statName] = character[statName] + buffValue
    end
  end
  return character
end
Using pairs on the buffs table visits each buff key. Bracket notation — character[statName] — is required here because statName is a variable, not a literal key name. The nil check ensures we only buff stats that already exist on the character rather than creating new unexpected keys.
F1 — The colon syntax / self
RELICS_LABYRINTH.Register( RELICS_LABYRINTH, thisUnit )
The colon is shorthand that automatically passes the table itself as the first argument. So self inside Register is the RELICS_LABYRINTH table — not the unit, and not some special keyword. It's just a regular parameter that Lua fills in silently. That's how self.relics inside the function reaches back into the table's own data.
F2 — Dot vs. bracket notation
Correct line: relic[propKey] = newValue
relic.propKey tells Lua to look for a key literally named "propKey" on the relic — which almost certainly doesn't exist. What we want is to use the value of the variable propKey as the key name. Bracket notation does exactly that: relic[propKey] reads the variable, gets its string value (e.g. "altarPoint"), and uses that as the key. This is the core reason bracket notation exists.
F3 — Missing return after remove
Problem: after table.remove shifts all entries left, the loop continues with a stale index and may skip the entry that shifted into position i, or run off the end of the table.
Fix: add return immediately after table.remove. Once you've found and removed the target, there's no reason to keep looping — and continuing is actively dangerous because the table's contents have just shifted. 
if relic.relicName == relicName then
  table.remove(self.relics, i)
  return   -- stop immediately
end
F4 — Sample Answer
ACTIVE_BUFFS = {
  buffs = {},

  Add = function(self, buffName, duration)
    local entry = {
      name     = buffName,
      duration = duration
    }
    table.insert(self.buffs, entry)
  end,

  Remove = function(self, buffName)
    for i, buff in ipairs(self.buffs) do
      if buff.name == buffName then
        table.remove(self.buffs, i)
        return
      end
    end
  end,
}
Things to check in your answer: comma after the Add function's closing end; return after the remove to stop looping; colon syntax used when calling (ACTIVE_BUFFS:Add("Haste", 10)). The Remove function silently does nothing if the buff name isn't found — that's fine and intentional.
